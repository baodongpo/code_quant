## 一、确定性 Bug（必须修复）

### Bug 1：`KlinePushHandler` 继承错误 —— 运行时必崩

**文件**：`futu/subscription_manager.py:37`

```python
class KlinePushHandler:
    def on_recv_rsp(self, rsp_pb):
        ret, content = super().on_recv_rsp(rsp_pb)  # ← 调用 object.on_recv_rsp，不存在
```

`KlinePushHandler` 没有继承富途 SDK 的任何基类（应继承 `SortedKlineHandlerBase` 或 `KlineHandlerBase`），但 `on_recv_rsp` 里直接调用了 `super().on_recv_rsp(rsp_pb)`。`object` 没有这个方法，运行时会抛 `AttributeError`，实时推送路径完全失效。

**另外**，`SubscriptionManager.register_push_handler()` 在 `main.py` 从未被调用，推送 handler 实际上没有注册到 `OpenQuoteContext`，即便修复了继承问题，实时推送也不会触发。

---

### Bug 2：`_heal_gaps` 中 `status='failed'` 的空洞无法被重试

**文件**：`core/sync_engine.py:217` + `db/repositories/gap_repo.py:19`

`get_open_gaps` 只查 `status='open'`，`status='failed'` 的空洞下次运行会重新被检测到，然后 `upsert_gaps` 使用 `INSERT OR IGNORE`，**不会更新已有的 failed 记录为 open**，导致 failed 空洞永远无法被重试修复——它既不在 open 列表里被处理，也不能被重置。

---

### Bug 3：`KlineFetcher.fetch()` 分页逻辑错误 + 全量同步数据截断

**文件**：`futu/kline_fetcher.py:47-80`

```python
next_time = start_date
while True:
    ret, data, next_page_req_key = self._client.ctx.request_history_kline(
        ...
        page_req_key=None if next_time == start_date else next_page_req_key,
    )
    ...
    if next_page_req_key is None:
        break
    next_time = next_page_req_key  # ← 把 page_req_key 赋给了 next_time（日期字段），语义混乱
```

更关键的是：**`fetch()` 根本没有被调用**——`SyncEngine._fetch_and_store` 和 `_heal_gaps` 均调用 `fetch_simple()`（非分页版本），而 `fetch_simple()` 传入 `max_count=_PAGE_SIZE=1000` 但没有翻页处理。对一只从 2015 年至今的日K数据（约 2500 条），**数据将被截断到最早的 1000 条**，且没有任何警告日志。

---

### Bug 4：`invalid_bars` 未写入数据库，违背需求

**文件**：`core/sync_engine.py:260-267`

需求文档 §3.5 明确要求：
> "校验失败的 bar 标记 `is_valid=0`，不直接丢弃"

但 `_fetch_and_store` 对 `invalid_bars` 只打 WARNING 日志，没有将它们以 `is_valid=0` 写入 `kline_data`。这违背了需求，且 `validator.py:37` 里的 `object.__setattr__` 调用也是死代码（因为无效 bar 最终没有被写入）。

---

### Bug 5：`SyncMetaRepository.upsert` 中 `rows_fetched/rows_inserted` 累加语义错误

**文件**：`db/repositories/sync_meta_repo.py:40-41`

```sql
rows_fetched  = sync_metadata.rows_fetched + excluded.rows_fetched,
rows_inserted = sync_metadata.rows_inserted + excluded.rows_inserted,
```

每次同步都在**累加**总行数，而不是记录"本次同步的行数"。几十次同步后，这两个字段完全失去观测意义。需求 §4 要求可观测，字段语义应是"最近一次同步行数"。

---

### Bug 6：`init_db` 中 `os.makedirs` 的 edge case

**文件**：`db/schema.py:153`

```python
os.makedirs(os.path.dirname(db_path), exist_ok=True)
```

若 `db_path` 是纯文件名（如 `"quant.db"`，无目录部分），`os.path.dirname` 返回 `""`，`os.makedirs("")` 会抛 `FileNotFoundError`。虽然默认值是 `data/quant.db`，但这是一个隐性炸弹。

## 二、设计一致性缺陷

### 缺陷 1：`fetch_simple()` 缺少分页，全量同步被截断

`SyncEngine._fetch_and_store` 和 `_heal_gaps` 都调用 `fetch_simple()`，而 `fetch_simple()` 写死了 `max_count=1000`，且没有翻页。对一只从 2015 年至今的日K数据（约 2500 条），**数据将被截断到最早的 1000 条**，且没有任何日志告警。（与 Bug 3 关联）

---

### 缺陷 2：`_sync_one` 的 `first_sync_date` 在强制全量时不更新

**文件**：`core/sync_engine.py:128-133`

```python
self._sync_meta_repo.upsert(
    ...
    first_sync_date=start_date if (force_full or meta is None) else None,
)
```

`SyncMetaRepository.upsert` 使用 `COALESCE(sync_metadata.first_sync_date, excluded.first_sync_date)`，即只有 DB 中没有值时才写入。当 `force_full=True` 重新全量同步时，`first_sync_date` 不会被更新（因 DB 已有值），可能导致重激活场景的起始日不准确。

---

### 缺陷 3：`AdjustFactorRepository.upsert_many` 使用 `INSERT OR IGNORE`，错误因子无法修正

**文件**：`db/repositories/adjust_factor_repo.py:12`

注释写"不修改历史"，设计意图正确。但如果某次除权因子数据从富途拉取时出现问题（比如 `forward_factor=1.0` 的默认值），`INSERT OR IGNORE` 会使错误数据永久留在库中无法修正。

---

### 缺陷 4：`AdjustFactorFetcher` 中手动计算分支是不完整实现

**文件**：`futu/adjust_factor_fetcher.py:67-73`

```python
# 此处简化为 1.0（需根据实际 SDK 字段调整）
fwd = 1.0
bwd = 1.0
```

若实际 SDK 没有返回 `forward_factor` 列，所有复权因子均为 1.0，前复权计算完全失效，且只有一条 WARNING 日志，不会报错。这是一个**静默错误**。

---

### 缺陷 5：`settings.py` 中 `DEFAULT_HISTORY_START` 默认值与需求文档不符

**文件**：`config/settings.py:38`

```python
DEFAULT_HISTORY_START = os.getenv("DEFAULT_HISTORY_START", "2015-01-01")
```

需求文档 §3.1 和设计文档 §4.3 均写明起始日为 **"2000-01-01"**，但代码默认值是 `2015-01-01`，`.env.example` 也是 `2015-01-01`。虽然可配置，但默认值与文档不一致。

---

### 缺陷 6：`KlineValidator.validate()` 单条接口是死代码

**文件**：`core/validator.py:58`

`validate()` 方法只在类内定义，未被任何调用方使用，属于 dead code。

---

### 缺陷 7：`data_gaps` 表没有唯一约束，导致重复空洞记录

**文件**：`db/schema.py:102-114`

`data_gaps` 表只有自增 `id` 作为主键，`(stock_code, period, gap_start, gap_end)` 没有唯一约束。`gap_repo.upsert_gaps` 使用 `INSERT OR IGNORE`，但由于没有唯一约束，IGNORE 永远不会触发，**每次检测到同一个空洞都会插入新行**，导致重复记录不断累积。

## 三、需要确认的疑点

| # | 文件 & 行号 | 疑点描述 |
|---|------------|---------|
| Q1 | `futu/kline_fetcher.py:52` | `fetch()` 使用 `AuType.QFQ`，`fetch_simple()` 使用 `AuType.NONE`。两个接口复权参数不一致，是否有意为之？ |
| Q2 | `futu/subscription_manager.py:162` | 注释说"仅日K推送，周K/月K通过历史接口"，但实际循环 `for period in ALL_PERIODS` 订阅了三个周期。逻辑与注释矛盾，以哪个为准？ |
| Q3 | `core/adjustment_service.py:87` | 注释提到"富途 forward_factor 是累乘值"——若富途直接返回的是**累积因子**（而非单次因子），则对所有 `ex_date > t` 的因子直接相乘是**错误的**（会重复累乘）。需要确认富途 `get_rehab` 返回的究竟是单次因子还是累积因子。这是前复权计算正确性的核心问题。 |
| Q4 | `core/sync_engine.py:172` | `_ensure_calendar` 将 `calendar_fetcher.fetch` 传入 `rate_limiter.execute_with_retry`，日历拉取也会受限频控制。需求文档说限频仅作用于 `get_history_kline`，是否应将日历请求排除在限频之外？ |
| Q5 | `main.py:153-155` | `active_stocks` 为空时在 `try` 块内 `return`，`finally` 会正常执行 `disconnect()`，无资源泄漏。但此时跳过了 `sync_subscriptions`，已有的活跃订阅不会被取消。若用户清空 watchlist 后重启，OpenD 中的订阅会残留。 |
